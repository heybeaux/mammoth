## Purpose

This document is the restart point for the post-MVP release train after P3. It
records the proved baseline, the intended P4-P6 sequence, likely ownership
boundaries, acceptance strategy, and operational lessons from P3.

In this document, **P4, P5, and P6 mean the checkpoints in
`POST_MVP_ROADMAP.md`**, not the similarly numbered phases in section 42 of
`ARCHITECTURE.md`. The roadmap checkpoints are:

1. P4 — research-cell contracts and lineage;
2. P5 — isolated divergence and blind review;
3. P6 — broader research topology.

The architecture remains normative. This handover proposes implementation slices,
package placement, verifier names, and release labels; a new `P4_PLAN.md` must
freeze those details before implementation.

## Architectural boundaries to preserve

These constraints survived P1-P3 and remain release gates:

- Models and research cells propose. Deterministic services validate and commit.
- Agreement, panel size, confidence, or persuasive prose never promotes truth.
- Every factual report sentence resolves to an admitted claim, a named policy
  verdict, an exact locator, and an immutable evidence digest.
- Criterion drift, dissent, contradictions, failed work, red results, and partial
  execution remain inspectable.
- Model memory, Temporal history, and Observatory projections are not product
  authority.
- External effects use stable idempotency keys and attributable receipts.
- Cloud egress requires classification, policy, provider, budget, and cost data.
- Adapters communicate through inward-facing ports or domain events. Do not add
  direct cross-adapter imports.
- UI and projection packages are read-only and non-authoritative.
- Workflow code must remain deterministic. Network, model, persistence, clock,
  and side-effect work belongs in Activities or ports.

Two kinds of lineage must remain distinct:

- `SourceLineage` already exists in `packages/domain/src/lineage.ts` and describes
  evidence/source derivation.
- P4 model lineage describes provider, model family, checkpoint, fine-tune,
  shared derivation, and correlation. Do not overload `SourceLineage` or reuse its
  table names for models.

## Existing package seams

The current codebase already exposes useful foundations:

- `packages/domain`: criteria, claims, evidence, source lineage, programs, audit,
  receipts, canonical digests, and pure state transitions.
- `packages/evidence`: evidence-policy and audit primitives.
- `packages/workflow`: durable workflow types plus the frozen P3 contract,
  workflow IDs, versioning, signals, queries, replay, and `continueAsNew` carry.
- `packages/governance`: budget ledger, human gates, revalidation, audit journal,
  and local governance persistence.
- `packages/work-queue`: leases, retry classification, stable effect identities,
  and completed-effect receipts.
- `packages/persistence` and `packages/postgres-adapter`: authoritative ledger
  ports, migrations, transactions, outbox, work state, and Activity effects.
- `packages/temporal-adapter`: P3 production Activities, research workflow,
  worker lifecycle, operator client/CLI, replay, and recovery probes.
- `packages/adapter-contracts`: versioned descriptors, capability negotiation,
  fail-closed startup, and behavioral conformance suites.
- `packages/observatory-projection`: deterministic read-only program, dossier,
  audit, and Temporal execution projections.
- `packages/report-compiler`: evidence-bound report-manifest validation and
  rendering; it cannot introduce facts.
- `apps/cli`: stateless operator entry point.
- `evals`: deterministic fixtures, checkpoint verifiers, and receipts.

Prefer extending these seams over creating a generic "agents" package. If a new
package is justified, name its authority and dependency direction in an ADR first.

## P4 — Research-cell contracts and lineage

### Outcome

Introduce deterministic, versioned contracts for research cells and model
lineage without yet claiming peer-isolated execution. P4 must make correlation,
self-review, criterion drift, and missing provenance mechanically rejectable.

### Required domain surface

Freeze schemas for at least:

- cell plan and cell-template version;
- typed cell input and required output contract;
- research position;
- review and review assignment;
- dissent/minority report;
- synthesis artifact restricted to admitted claim IDs;
- model profile and immutable model-profile version;
- provider, family, checkpoint, fine-tune, training lineage, shared derivation,
  locality, modalities, context window, data policy, and cost profile;
- correlation assessment with explicit reasons and policy version;
- cell result, failure, cancellation, and receipt references.

Every position and review must carry immutable `programId`, `workItemId`,
`criterionId`, criterion digest/version, model-profile ID/version, input digest,
output schema version, usage, uncertainty/failure codes, and proposal references.
Unknown model lineage is represented as unknown and receives a conservative
correlation penalty; it is never treated as independent.

### Suggested P4 slices

#### P4-T1 — Contract freeze

Own `packages/domain`, relevant architecture excerpts, and a new ADR. Add pure
Zod schemas and canonical identity/digest functions. Keep persistence, Temporal,
and provider SDKs out of the domain package.

Acceptance should reject unknown schema versions, mutable criterion references,
cyclic model lineage, missing parents, malformed shared-derivation data, and
non-canonical digests.

#### P4-T2 — Deterministic admission and correlation policy

Implement pure admission rules for positions and reviews. A review must fail when
the same agent/model profile reviews its own candidate in the same role. Related
profiles must not be counted as independent merely because their public model IDs
differ.

The policy must reject missing claim/evidence/hypothesis/artifact references,
criterion drift, unresolved model lineage where the policy requires independence,
and synthesis over non-admitted claims. It must preserve rejected proposals and
reasons for audit rather than deleting them.

#### P4-T3 — Authoritative persistence

Add forward-only Postgres migrations and ports for model profiles and versions,
cell plans, positions, reviews, dissent, correlation assessments, and receipts.
Test empty install, upgrade, interrupted migration, uniqueness, optimistic
revision/fencing, foreign keys, and restart.

Store immutable model-profile versions. Never rewrite historical lineage when a
provider alias or marketing name changes.

#### P4-T4 — Application and workflow boundary

Add inward-facing ports and versioned workflow/application contracts for cell
plans and work items. Stable IDs must include program, criterion version, cell
plan version, branch, and role where appropriate. P4 may prove contract carriage
and reconstruction without claiming P5 peer isolation.

Product state must reconstruct from Postgres/CAS after `continueAsNew`; do not
grow Temporal carry state into a shadow product store.

#### P4-T5 — Projection and operator inspection

Extend Observatory projections with read-only cell, position, review, model
lineage, correlation, dissent, and receipt nodes/edges. The projection builder
must fail closed on future authority, broken references, digest mismatch, or
lineage cycles.

Add CLI inspection only after the projection contract is stable. Do not build the
3D UI as part of this checkpoint.

#### P4-T6 — Acceptance and receipt

Add `pnpm verify:p4`, a deterministic fixture suite, a clean-checkout receipt, and
default-branch CI coverage. A proposed release label is
`v0.4.0-research-cell-contracts`; freeze the actual label in `P4_PLAN.md`.

### Minimum P4 adversarial fixtures

- one unsupported claim with one agreeing position;
- the same unsupported claim with 100 agreeing positions;
- two aliases of the same checkpoint/family presented as independent;
- unknown lineage presented as independent;
- a model reviewing its own candidate under the same role;
- a distinct model from a correlated lineage reviewing a candidate;
- valid cross-family review;
- silent criterion edit and explicit criterion branch;
- missing claim/evidence references;
- cyclic or dangling model lineage;
- rejected position retained with its audit reason;
- deterministic projection digest and restart reconstruction.

The P4 exit proof is not "schemas exist." The verifier must prove that arbitrary
agreement cannot promote an unsupported claim and that self-review is impossible
under the frozen policy.

## P5 — Isolated divergence and blind review

### Outcome

Execute research positions in isolated divergence cells, commit them before peer
exposure, assign blind independent reviews, preserve dissent, and settle budgets
and cancellation honestly.

### Suggested P5 slices

1. **Isolation protocol:** freeze commit-before-reveal semantics. A committed
   position has an immutable input digest, position digest, criterion reference,
   model profile, and audit sequence before peer material is released.
2. **Sanitized review context:** omit author identity, author model/provider,
   confidence, candidate popularity, previous verdicts, and upstream PASS markers
   from reviewer input while retaining them in authoritative audit state.
3. **Assignment policy:** deterministically prohibit same-role self-review and
   penalize or reject correlated panels according to a versioned policy.
4. **Dissent and residue:** retain minority reports, unresolved conflicts,
   criterion drift, abstentions, invalid reviews, and uncompleted assignments.
5. **Budget lifecycle:** reserve budget per cell before dispatch; record attempted,
   afforded, rejected, consumed, released, and cancelled amounts. A cancelled or
   timed-out cell returns honest partial results and receipts.
6. **Temporal execution:** implement divergence/review workflows and Activities
   with stable IDs, idempotency, signals, cancellation, replay, versioning, and
   bounded history.
7. **Recovery and observability:** kill clients, workflow workers, Activity
   workers, and the local Temporal service before and after commit/reveal,
   assignment, review commit, budget settlement, and cancellation boundaries.
8. **Acceptance:** add `verify:p5`, offline fixtures, a checkpoint receipt, and a
   clean-checkout/default-branch proof. A proposed label is
   `v0.5.0-isolated-divergence`.

### Minimum P5 exit evidence

- A worker cannot read peer positions before its own position is durably committed.
- Repeated delivery cannot duplicate a position, review, reservation, release, or
  provider charge.
- Reviewer input contains no prohibited author/popularity/verdict fields.
- The assignment record and audit log still retain attribution.
- Same-role self-review is rejected; correlation affects panel independence.
- Minority and unresolved positions survive synthesis, restart, and cancellation.
- Spend never exceeds the reservation/policy ceiling.
- Cancellation returns attributable partial results and receipt references.
- Replay succeeds across every supported workflow version.

Do not use process scheduling order or an in-memory flag as proof of isolation.
Isolation must be visible in authoritative commits, digests, and audit sequence.

## P6 — Broader research topology

### Outcome

Compose the P4/P5 primitives into a restart-safe, budget-bounded multi-cell
program covering landscape, divergence, prior art, falsification, experiment,
and synthesis work.

### Required topology templates

Implement versioned templates corresponding to architecture section 18:

- Landscape: retriever → claim extractor → lineage analyst;
- Divergence: multiple independent lateralists, blind to peers;
- Prior art: literature scout → patent/code scout → novelty challenger;
- Falsification: empiricist → adversary → boundary-condition analyst;
- Experiment: experiment designer → evaluator author → reproduction worker;
- Synthesis: evidence compiler → dissent reporter.

Cells exchange typed IDs, artifacts, contracts, and receipts—not unconstrained
full transcripts.

### Suggested P6 slices

1. Freeze topology-plan, dependency-edge, concurrency, budget, and failure-policy
   contracts. Reject cycles, missing nodes, invalid template versions, and
   unbounded concurrency.
2. Build a deterministic planner whose decisions are versioned and inspectable.
   It must distinguish idle work from budget-starved work.
3. Add Temporal parent/child workflow composition with stable IDs, bounded
   histories, `continueAsNew`, cancellation propagation, and partial-result
   collection.
4. Implement evidence-aware synthesis over admitted claim IDs only. Vote count
   and cell consensus remain descriptive metadata, never truth authority.
5. Add read-only projections for topology, cell state, model lineage,
   disagreement, spend, receipts, retries, and partial failure.
6. Add offline fixtures for correlated consensus, criterion drift, missing
   evidence, adversarial dissent, budget starvation, worker/service death, and
   partial failure.
7. Add `verify:p6`, a complete multi-cell acceptance run, receipt, CI gate, and
   proposed `v0.6.0-research-topology` label.

### P6 stopping condition

A complete multi-cell program must survive process and service restart, preserve
dissent, remain within budget, retain failed/partial work, and emit a dossier in
which every factual sentence still satisfies the original MVP provenance gate.

P6 does not automatically include the architecture's later novelty archive,
production experiment sandbox, Engram/AWM adaptation, governance-hardening UI,
or `mammoth-pipelines` work unless a separately approved plan moves those items
into scope.

## Agent-runtime gotchas and proof rules

P3 exposed a serious coordination failure: workers were repeatedly described as
active when they were not. Treat the following as hard operating rules.

### There are two delegation runtimes

- OpenClaw subagents (`sessions_spawn`/the OpenClaw subagent registry) are durable
  and observable across Telegram turns.
- Codex-native agents (`spawn_agent`/`list_agents`) belong to one Codex root
  rollout. A later Telegram turn creates a new root that cannot reliably observe
  or manage the old native tree.

Never use one registry as evidence about the other runtime. For a detached,
multi-turn loop, use OpenClaw and retain the child session key and run ID. Use
Codex-native workers only when the parent turn will remain open through completion.

### Spawn acceptance is not liveness

Use these exact states:

- **spawn requested:** the call was made or accepted;
- **active:** the matching live registry reports `running`;
- **producing:** the live worker also has a fresh, attributable artifact, diff,
  test result, or progress report;
- **completed:** it returned a handoff;
- **integrated:** the coordinator reviewed and incorporated the work;
- **merged/verified:** default branch and CI prove it.

A prepared worktree, assignment text, branch, commit from an older run, or
`status: accepted` is not evidence of an active worker.

Before telling Beaux a worker is active, require all of:

1. the same runtime's live registry identifies the exact session/run;
2. status is `running`, not accepted, idle, failed, timed out, or missing;
3. the worker has emitted a fresh timestamped liveness marker or progress message;
4. for implementation work, a tied artifact/diff/test appears after spawn.

### Model-routing failures seen during P3

- The initial OpenClaw coordinator used Claude Sonnet 4.6 and failed almost
  immediately because the configured Claude accounts were out of usage.
- Switching Claude accounts made a small probe succeed, but that did not prove a
  durable coordinator was running.
- Initial Claude Opus 4.8 coordinator/probe attempts timed out without useful
  output. A later exact Opus 4.8 coordinator did complete a long clean-checkout
  audit and caught the missing default-branch `verify:p3` CI step. Treat each run
  independently; a later success does not retroactively prove an earlier spawn
  was active.
- A nominal Terra coordinator routed to GPT-5.5, then failed because inherited
  `thinking: high` was unsupported. A `thinking: off` retry still routed to
  GPT-5.5.
- The proven route was exact `openai-codex/gpt-5.6-terra` with thinking off. It
  produced a live registry entry, a timestamped marker, and a fresh commit.
- Beaux explicitly required subagents to be moved off Sonnet 4.6. Do not put P4-P6
  workers back on it without new instruction.

Model aliases, configured defaults, and the name in an assignment are not proof of
the model that actually ran. Inspect the live session's resolved model.

### Prevent orphaned and overlapping workers

Status questions caused new Codex roots to spawn replacement teams while older
native workers continued in the same worktrees. Four roots launched twelve
workers across three paths. This created overlapping writers and contradictory
status reports.

- Never respawn because someone asks for status. First prove the old run is dead.
- One worker owns one worktree/path at a time.
- Before replacement, inspect live registries, recent file timestamps, diffs,
  process state, and session history.
- A timed-out control request may leave a worker or its output alive. Inspect the
  worktree before assigning a replacement.
- Kill or retire stale registry entries before launching the replacement.
- Keep one execution slot free for coordination and integration.
- Require bounded assignments with owned paths, non-owned paths, permitted
  contract changes, exact verification, and handoff recipient.

### Recommended launch protocol

1. Spawn one bounded probe on the intended exact model.
2. Verify resolved model, live `running` state, timestamped output, and a harmless
   artifact.
3. End the probe and verify its completed state.
4. Spawn one real bounded lane.
5. Prove it end-to-end before distributing more lanes.
6. Record session key, run ID, model, worktree, branch, owned paths, and last
   verified output in the coordinator ledger.
7. Poll the durable registry at useful boundaries. Do not claim ongoing activity
   from stale evidence.

## Repository and integration gotchas

- The authoritative repository is `heybeaux/mammoth`. The clean P3 integration
  worktree was `/private/tmp/mammoth-p3-t1-integration`; future work should create
  a fresh checkpoint worktree from current `origin/main`.
- At the end of P3, the `beaux-riel` GitHub CLI account could read the repository
  but lacked merge permission. `heybeaux` could merge. Check `gh auth status`
  before a release operation, switch only when needed, and restore the prior
  account afterward.
- A commit created in `/Users/beauxwalton/mammoth` may not exist in a worktree
  backed by `/Users/beauxwalton/projects/mammoth`. Fetch the alternate clone path
  explicitly before cherry-picking by SHA.
- Warm worktrees can hide missing ignored `dist/` output. Verify dependency builds
  from a clean checkout; do not assume pnpm runs an implicit pretest build.
- P2 Postgres/CAS lifecycle and backup verification must remain independently
  runnable. Routing it through a Temporal-required combined profile caused
  `spawn temporal ENOENT` and was a regression, not a P2 failure.
- Temporal SDK CLI bundles must prebuild the temporal adapter sub-exports and mark
  `@temporalio/*` and `@grpc/*` external. Bundling them into CJS caused dynamic
  `require` failures.
- `.receipts/` and OpenClaw session artifacts must not enter product commits or
  Prettier scope. Stage owned paths explicitly; never `git add .`.
- Temporal history is orchestration evidence. Recovery is not complete until it
  reconciles with Postgres/CAS/audit/outbox/receipt authority.
- Keep real local-service fault tests separate from pure contract tests so an
  unavailable external binary cannot turn a deterministic unit gate into noise.
- Do not weaken or delete a verifier to make CI green. Preserve red evidence and
  change strategy after repeated identical failures.
- A local checkpoint gate is not enough. Confirm the corresponding `verify:pN`
  command appears as an actual default-branch CI step and inspect the green run
  after merge. P3 needed PR #32 specifically because this was initially missed.

1. reconcile P3-as-active language;
2. write `P4_PLAN.md` and the P4 loop;
