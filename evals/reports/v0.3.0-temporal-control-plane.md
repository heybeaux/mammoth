## Scope

This checkpoint moves Mammoth's evidence-first MVP program onto a Temporal-backed
control plane while preserving the P2 authority boundary: Postgres and CAS remain
authoritative for product state, artifacts, receipts, audit, outbox, and
read-only projections. Temporal owns workflow history, signals, queries, timers,
retries, cancellation delivery, and orchestration replay only.

The implementation covers P3 T2 through T6 on top of the previously merged T1
contract and lifecycle foundation. It does not claim managed Temporal deployment,
research-cell fan-out, or the 3D Observatory UI.

## Temporal adapter descriptor and lifecycle

The adapter contract remains under major `1`. The Temporal runtime capability is
described through the workflow-runtime descriptor and fail-closed readiness
checks in `@mammoth/adapter-contracts` and `@mammoth/temporal-adapter`.

Local and CI lifecycle are implemented by `@mammoth/temporal-adapter`:

- namespace: configured by `MAMMOTH_TEMPORAL_NAMESPACE`, defaulting to the local
  Mammoth namespace in package tests;
- task queue: configured by `MAMMOTH_TEMPORAL_TASK_QUEUE`, with the research
  control workflow using `mammoth-research-control-v1` in the operator examples;
- service: operator-installed Temporal CLI `server start-dev`, no embedded
  credentials;
- readiness: service reachability, namespace/retention validation, task-queue
  and worker-manifest checks, and live SDK capability probes;
- shutdown: bounded stop with PID/start-time/command ownership checks to avoid
  signalling an unrelated process.

## Workflows, versions, and IDs

The frozen workflow contract is `P3_WORKFLOW_CONTRACT_MAJOR = 1`.

Supported workflow definitions are recorded in
`packages/workflow/src/p3-contract.ts`:

- `ResearchProgramWorkflow` v1 on `research-control`;
- `AcquisitionWorkflow` v1 on `retrieval`;
- `HypothesisCampaignWorkflow` v1 on `local-large`;
- `ExperimentWorkflow` v1 on `experiment`;
- `RevalidationWorkflow` v1 on `retrieval`;
- `ReportCompilationWorkflow` v1 on `research-control`;
- `HumanReviewWorkflow` v1 on `human-gate`.

The implemented MVP Temporal workflow is `researchProgramWorkflow` in
`packages/temporal-adapter/src/research-workflows.ts`. Its workflow ID is derived
from the frozen workflow name/version plus `{ programId, criterionVersion,
branchId }`; it rejects a run whose Temporal workflow ID does not match that
stable identity. Version 1 records the Temporal patch marker
`research-program-v1-stage-order`.

The v1 MVP stage order is:

1. `commit-budget`
2. `snapshot-source`
3. `assess-claims`
4. `persist-ledger`
5. `compile-report`
6. `commit-receipt`

The `continueAsNew` policy is `maxCyclesPerRun = 1` and
`maxHistoryEvents = 128`. The carry set is intentionally narrow:
workflow ID/kind/version plus program, criterion-version, and branch IDs. Product
state is reconstructed through Activities from authoritative stores.

## Signals, queries, cancellation, and human gates

The v1 signal surface is `researchProgram.control.v1` with guarded `pause`,
`resume`, `cancel`, `criterion-branch`, and `human-gate-decision` messages.
Signals carry a stable `signalId` and `expectedRevision`; duplicate and stale
signals do not advance control state.

The query surface is:

- `researchProgram.state.v1`;
- `researchProgram.step.v1`;
- `researchProgram.gates.v1`;
- `researchProgram.cancellation.v1`;
- `researchProgram.retry.v1`;
- `researchProgram.receipts.v1`.

Human-gate specifications are persisted before the first stage, decisions are
recorded through an Activity, and timeout/rejection/cancellation return partial
results with existing receipt references only.

The stateless operator CLI is `mammoth-temporal-research`, with tested `run`,
`status`, `pause`, `resume`, `cancel`, `inspect`, `approve`, and `reject`
commands from separate processes.

## Activities and idempotency receipts

The P3 Activity catalog is implemented in
`packages/temporal-adapter/src/production-activities.ts` and backed by the
work-queue effect contract plus Postgres effect store:

- `retrieval`
- `snapshot`
- `parsing`
- `claim-proposal-admission`
- `assessment`
- `ledger-mutation`
- `report-compilation`
- `artifact-commit`
- `outbox-publication`
- `revalidation`
- `human-gate-handoff`

Provider idempotency keys are derived from stable Activity invocation inputs and
the attributable work item, matching the architecture formula
`programId + workItemId + contractVersion + inputDigest + operationKind`.
Before retryable side effects, Activities check the completed-effect store and
return the existing receipt instead of repeating the provider effect. Retry,
heartbeat, timeout, and non-retryable failure classes are frozen by
`activityPolicies` and `nonRetryableActivityFailureCodes` in
`@mammoth/work-queue`.

## Crash, restart, and recovery

Recovery evidence is implemented in the Temporal adapter live and recovery test
suites and summarized in `docs/P3_RECOVERY.md`.

The recovery probes cover:

- workflow poller loss and replacement worker completion;
- client/CLI process kill while awaiting a workflow result;
- separate workflow-worker and Activity-worker process kill boundaries;
- forced duplicate Activity delivery after a simulated provider commit;
- stale signal handling;
- heartbeat timeout and retry exhaustion coverage;
- persistent Temporal service stop/start using the configured local persistence
  path;
- workflow history download and replay with the recorded workflow bundle.

The documented production rule remains that Temporal recovery must be reconciled
against Postgres/CAS/audit/outbox/receipt authority before operators mark product
state recovered.

  the post-merge default branch CI run is the final confirmation gate.

  the product recovery source of truth.

  cannot be satisfied without operator-supplied credentials.
